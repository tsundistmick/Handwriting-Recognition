# TrOCR Baseline

## Как устроен TrOCR внутри

TrOCR — это нейросетевая модель от Microsoft, построенная на архитектуре **трансформера**. В отличие от Tesseract, который обрабатывает изображение через последовательность независимых этапов, TrOCR обучен **end-to-end**: он напрямую преобразует изображение строки в текст, без промежуточных шагов сегментации символов или словарной коррекции.

### Общий пайплайн

```
Изображение строки
    ↓
1. Нормализация и разбиение на патчи (Processor)
    ↓
2. Vision Encoder (ViT) — понимает изображение
    ↓
3. Text Decoder (BEiT/RoBERTa) — генерирует текст
    ↓
4. Beam Search — выбирает лучшую последовательность
    ↓
Текст
```

---

### Компонент 1 — Processor (предобработка)

Перед подачей в нейросеть изображение проходит через `TrOCRProcessor`. Он делает две вещи:

**Обработка изображения.** Изображение масштабируется до фиксированного размера (384×384 пикселя), нормализуется по среднему и стандартному отклонению ImageNet, затем разбивается на патчи 16×16 пикселей. Из изображения 384×384 получается сетка 24×24 = 576 патчей.

**Токенизация текста.** Для обучения текст преобразуется в токены через BPE (Byte-Pair Encoding) — тот же принцип, что в GPT. При инференсе токенизатор нужен для декодирования выходных токенов обратно в строку.

---

### Компонент 2 — Vision Encoder (ViT)

Encoder — это **Vision Transformer (ViT)**. Каждый патч 16×16 пикселей превращается в вектор размерности 768 через линейную проекцию. К этим векторам добавляется позиционное эмбеддинг — информация о том, где в изображении находится каждый патч.

Затем последовательность из 576 векторов проходит через стек трансформерных блоков. Каждый блок состоит из:

- **Self-attention** — каждый патч смотрит на все остальные патчи и понимает контекст
- **Feed-forward сети** — нелинейное преобразование каждого вектора отдельно

Один блок = self-attention + feed-forward. В TrOCR-base таких блоков 12 штук, они идут последовательно:
```
576 патчей
    ↓
[Блок 1: self-attention → feed-forward]
    ↓   патчи начинают понимать ближайших соседей
[Блок 2: self-attention → feed-forward]
    ↓   патчи видят более широкий контекст
[Блок 3: self-attention → feed-forward]
    ↓
   ...
[Блок 12: self-attention → feed-forward]
    ↓
576 векторов по 768 чисел каждый
```
На первых блоках патчи понимают локальный контекст — соседние буквы. На последних блоках — глобальный: стиль почерка всей строки, наклон, толщину линий.
На выходе Encoder возвращает 576 контекстуализированных векторов — они кодируют визуальное содержание изображения.

```
576 патчей × 768 размерность → матрица [576, 768]
```

---

### Компонент 3 — Text Decoder

Decoder — это трансформер языковой модели (на базе BEiT или RoBERTa). Он генерирует текст **авторегрессивно**: предсказывает один токен за раз, используя все уже сгенерированные токены и выход Encoder.

Каждый шаг декодирования:

```
[уже сгенерированные токены] + [выход Encoder]
    ↓
Cross-attention: Decoder смотрит на выход Encoder
    ↓
Self-attention: Decoder смотрит на уже сгенерированный текст
    ↓
Линейный слой + Softmax → вероятности по всему словарю (~50000 токенов)
    ↓
Выбирается токен с наибольшей вероятностью
```

Генерация продолжается до тех пор, пока не будет выдан специальный токен конца строки `</s>`.

---

### Компонент 4 — Beam Search

Вместо того чтобы жадно брать самый вероятный токен на каждом шаге, TrOCR по умолчанию использует **Beam Search**: одновременно отслеживает несколько наиболее вероятных последовательностей и в конце выбирает лучшую.

---

### Почему TrOCR работает лучше Tesseract на рукописном

Tesseract обучался на синтетических печатных данных. TrOCR предобучен на **огромных датасетах рукописного текста** (IAM, SROIE, арXiv-рукописи) и дообучен специально на задаче HTR. Encoder учится видеть вариативность почерка, Decoder учится предсказывать правдоподобный текст — всё это оптимизируется совместно.

## Разбор кода

### Импорты и константы

`transformers` — библиотека HuggingFace, предоставляет готовые классы для загрузки и запуска предобученных моделей. `torch` нужен для работы с тензорами и управления устройством (CPU/GPU). Последняя строка импортирует вспомогательные функции из `tesseract_baseline.py` — чтобы не дублировать код парсинга датасета.

```python
MODEL_NAME  = "microsoft/trocr-base-handwritten"
BATCH_SIZE  = 8
DEVICE      = "cpu"
```

`MODEL_NAME` — идентификатор модели на HuggingFace Hub.  `BATCH_SIZE = 8` означает, что за один вызов модели обрабатывается 8 изображений одновременно — это быстрее, чем по одному

---

### Загрузка модели

```python
processor = TrOCRProcessor.from_pretrained(MODEL_NAME)
model     = VisionEncoderDecoderModel.from_pretrained(MODEL_NAME).to(DEVICE)
model.eval()
```

`TrOCRProcessor` объединяет в себе обработчик изображений и токенизатор текста.

`VisionEncoderDecoderModel` — это обёртка, которая соединяет Encoder и Decoder в одну модель. `.to(DEVICE)` переносит все веса на нужное устройство (CPU или CUDA).

`model.eval()` переключает модель в режим инференса. Это важно: в режиме обучения активен Dropout (случайно обнуляет часть нейронов для регуляризации), в режиме инференса он отключается. Без `model.eval()` результаты были бы случайными от запуска к запуску.

---

### `run_trocr_batch()` — обработка батча

```python
def run_trocr_batch(images: list, processor, model) -> list:
    pixel_values = processor(
        images,
        return_tensors="pt",
        padding=True
    ).pixel_values.to(DEVICE)

    with torch.no_grad():
        generated_ids = model.generate(pixel_values)

    texts = processor.batch_decode(generated_ids, skip_special_tokens=True)
    return texts
```

**`processor(images, return_tensors="pt", padding=True)`** — обрабатывает список изображений сразу. `return_tensors="pt"` означает вернуть PyTorch-тензоры. `padding=True` выравнивает изображения разной ширины до одного размера внутри батча — Tesseract обрабатывал каждое изображение отдельно, TrOCR может обрабатывать сразу несколько.

**`.pixel_values`** — извлекает только тензор пикселей из результата. Это тензор формы `[batch_size, 3, 384, 384]`: 8 изображений, 3 канала (RGB), 384×384 пикселей.

**`torch.no_grad()`** — отключает вычисление градиентов. Градиенты нужны только при обучении для обратного распространения ошибки. При инференсе они не нужны, но без этого контекста PyTorch всё равно их хранит, тратя память и время.

**`model.generate(pixel_values)`** — запускает авторегрессивную генерацию. Внутри происходит: прогон через Encoder, затем итеративный Decoder с Beam Search до появления токена конца строки. Возвращает тензор идентификаторов токенов.

**`processor.batch_decode(generated_ids, skip_special_tokens=True)`** — конвертирует токены обратно в строки. `skip_special_tokens=True` убирает служебные токены `<s>`, `</s>`, `<pad>` из результата.

### Батчевая обработка в `main()`

```python
batch_images = []
batch_refs   = []
batch_ids    = []

def process_batch():
    texts = run_trocr_batch(batch_images, processor, model)
    for line_id, ref, hyp in zip(batch_ids, batch_refs, texts):
        references.append(ref)
        hypotheses.append(hyp)
        writer_csv.writerow([line_id, ref, hyp])
```

`process_batch` определена внутри `main` — это **замыкание (closure)**. Она видит переменные внешней функции (`references`, `hypotheses`, `writer_csv`) напрямую, без передачи как аргументы. Эта функция нужна, чтобы прогнать уже полностью накопленный батч через модель.

```python
for i, line_id in enumerate(ids):
    ...
    batch_images.append(image)
    batch_refs.append(gt[line_id])
    batch_ids.append(line_id)

    if len(batch_images) == BATCH_SIZE:
        process_batch()
        batch_images.clear()
        batch_refs.clear()
        batch_ids.clear()

# Остаток после цикла
if batch_images:
    process_batch()
```

Изображения накапливаются в список. Когда набирается ровно `BATCH_SIZE` штук — вызывается `process_batch()` и списки очищаются. Последний блок `if batch_images:` обрабатывает остаток — финальный батч, в котором может быть меньше 8 изображений. Без него последние строки датасета были бы потеряны.

---

### Зачем батчевая обработка

Нейросеть устроена так, что обработка 8 изображений одновременно значительно быстрее, чем 8 отдельных вызовов. Причина в том, что операции над матрицами внутри трансформера хорошо параллелизуются.
